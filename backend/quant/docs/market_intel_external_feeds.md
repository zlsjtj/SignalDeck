# Market Intel External Feeds

This note defines how SignalDeck should connect news NLP and on-chain data without fabricating market context or committing private configuration.

## News NLP / Sentiment

Start with one of these real sources:

- RSS feeds from configured publishers, exchanges, research desks, or project blogs.
- A local feed endpoint controlled by the operator.
- A local file drop or queue generated by an existing news collection job.

Each raw item should be normalized before any NLP scoring:

- `id` or stable dedupe key.
- `source`.
- `url`.
- `title`.
- `summary` or body excerpt when available.
- `publishedAt`.
- Related assets or symbols when known.

NLP output must be labeled as an estimate, not fact:

- `sentimentScore`, for example `-1.0` to `1.0`.
- `confidence`, when the model or classifier provides one.
- `model` or classifier version.
- `scoredAt`.
- `reason` or short extracted rationale when available.

Frontend rules:

- Show source and publish time for every item.
- Show `source_not_configured` until a source is configured.
- Do not call sentiment a trading signal.
- Do not display generated or placeholder news.

## On-chain Data

Start with real aggregate sources:

- A chain data API or indexer configured by the operator.
- A self-hosted node plus an indexing job.
- A local aggregate feed generated by another internal service.

Prefer low-frequency aggregate storage before raw event ingestion:

- 5-minute or hourly exchange netflow.
- Stablecoin supply or transfer flow aggregates.
- Active address counts.
- Transaction count and fee or gas aggregates.
- Large transfer counts and notional buckets.

Each metric should include:

- `source`.
- `chain`.
- `asset`.
- `metric`.
- `value`.
- `windowSeconds`.
- `updatedAt`.
- `status`.
- `message` for missing or partial coverage.

Frontend rules:

- Show `source_not_configured` until an on-chain source is configured.
- Show freshness and coverage for every metric.
- Treat missing coverage as missing data, not zero.
- Present all on-chain metrics as monitoring context only.

## Secret Handling

- Keep API keys in server environment variables or private secret files.
- Do not commit `.env`, `.env.*`, `.secrets`, provider keys, exported CSVs, databases, logs, or local feed dumps.
- `.env.example` may document variable names with placeholder values only.

## Implementation Sequence

1. Add backend configuration for enabled feeds and source names.
2. Add a collector that normalizes source rows and records source status.
3. Store lightweight aggregates or recent normalized news rows.
4. Expose API payloads with `status`, `message`, `source`, `updatedAt`, and `rows`.
5. Update the market structure frontend to render configured rows and keep unconfigured states explicit.
